## 0. 這份框架要解決什麼

`kickoff.md` 把技術棧寫成「C++20 **或** Python + Numba/Cython」，這個「或」會連帶決定測試框架、CI 形態、以及 M4 效能目標是否可能達成。在 M0 的 spec 寫下去之前必須先收斂。

**設計前提**：評估者（Luke）已有預期答案是 Python。因此這個框架的價值**完全取決於它有沒有能力推翻那個預期**。若跑完必然得到 Python，那它只是事後合理化，不如不跑。

為此框架內建三個防呆機制：

1. **預先登記（pre-registration）** — 在看到任何分數之前，先寫下「什麼結果會讓我改選 C++/混合方案」（§6）。
2. **Gate 與加權分離** — 硬性門檻用 pass/fail，不可被便利性項目的高分沖掉（§3）。
3. **證據等級標註** — 每格分數註明是實測、推論、還是純判斷；Gate 只接受實測（§5）。

## 3. Gate 條件（pass / fail，先篩後評）

未通過 Gate 的方案**直接出局**，不進入加權評分。這是為了避免「開發速度 5 分 × 高權重」把一個達不到效能目標的方案抬進第一名。

### G1 效能可達性 — 門檻已重新推導

`kickoff.md` 的「50–100 agents 數十毫秒」**未交代推導來源**，且與專案自身的非目標矛盾：文件明訂「不處理動態即時避障」，代表本 planner 是**慢層（全域協調）**而非快層（反應式避障）。以快層的延遲預算要求慢層，等於為不存在的需求選技術棧。

改為從場景推導。推導只有一條式子：

```
延遲預算 = 網格邊長 ÷ 機隊速度
（若重規劃週期 < 單格通過時間，計畫落後現實永不超過一格）

網格邊長取 1 m：小型商用多旋翼翼展約 0.3–0.5 m，加安全餘裕
```

**設計決定：用階梯，不用單一門檻。** 評估者對無人機領域尚不熟悉，單一門檻是一堵牆——過不去就沒有中間狀態，等於用一個自己都不確定的數字把專案吊死。階梯讓入門門檻低到必過，同時每一階都對應真實場景。

| 階 | 場景 | 速度 | 延遲預算 |
|---|---|---|---:|
| **L0** | 室內倉儲、受限空間（安全考量壓速度） | 1 m/s | **1000 ms** |
| L1 | 室內低速巡檢 | 2 m/s | 500 ms |
| L2 | 室外低速 | 5 m/s | 200 ms |
| L3 | 室外巡航 | 10 m/s | 100 ms |
| **L4** | 高速巡航 | 15 m/s | **67 ms** |

> 速度為典型量級而非規格，待領域熟悉後回頭修正。**L4 約等於 `kickoff.md` 的「數十毫秒」——原始數字並非憑空而來，只是默默假設了最難的一階而未言明。**

| 層級 | 門檻 | 性質 |
|---|---|---|
| **G1 Gate** | 中型 benchmark（50–100 agents）達 **L0（< 1000 ms）** | 硬需求，未過出局 |
| **M4 目標** | 沿階梯往上推，目標 L3–L4 | M4 的驗收敘事從「有沒有達到數十毫秒」改為「把系統從 L0 推到第幾階」 |

判定方式：**必須實測**（§4 spike），不接受推論。

### G2 演算法表達力

能自然表達 CBS 的衝突樹遞迴結構與變長約束集合，不需為了語言限制扭曲演算法。以最小 CBS 骨架驗證。

> **特別針對 C 方案**：Numba nopython 對 dict of lists、遞迴樹狀結構、變長物件的支援有限。CBS 高階部分很可能無法 JIT，只有低階 A* 能。這不必然致命（高階節點數遠少於低階展開數），但必須驗證而非假設。

### G3 可重現性

CI 在乾淨機器上可建置並跑完測試，時間可接受。以建置腳本試跑驗證。

## 4. 讓 G1 從猜測變成測量：Space-Time A* micro-spike

G1 是整個決策的樞紐，也是最容易被含糊帶過的一格。關鍵在於把求解延遲拆成兩個可分別取得的因子：

```
求解延遲  ≈  每次求解的節點展開數  ×  每次展開的成本
             └─ 演算法決定，與語言無關 ─┘   └─ 語言決定 ─┘
```

- **右因子（每次展開成本）** — 由 spike 實測。各候選實作一份最小 Space-Time A*（狀態 `(node, timestep)`、支援 wait、binary heap、closed set），跑同一張 movingai 地圖的同一組單代理人問題，量測 **expansions/sec**。每個候選約半天。
- **左因子（展開數）** — 與語言無關，可用純 Python 版 CBS 在小型 benchmark 上跑一次、只計數不計時取得，再依 agent 數外推。

兩者相乘即得各候選的延遲估計，G1 隨之有 pass/fail 的實測依據。

**先驗（僅為待驗證的量級假設，非事實）**：CPython 的 A* 展開速率一般落在 10⁵ expansions/sec 量級，最佳化過的 C++ 可到 10⁷ 量級，Numba nopython 介於兩者、偏 C++ 側。若中型 benchmark 的 CBS 求解需要 10⁵–10⁶ 次展開：

- 對原始門檻（數十毫秒 ≈ L4）：B 幾乎確定不可行，A 不寬裕，C/D 在生死線上
- 對 Gate 門檻（L0 = 1000 ms）：預期四者皆過，Gate 退化為安全網而非篩選器

**這是階梯設計的刻意結果：Gate 不再負責淘汰，改由加權準則 #1（效能餘裕）承擔區分工作——問題從「誰能過關」變成「誰能爬到第幾階」。** spike 的產出因此不是 pass/fail，而是各候選的**可達階數**，資訊量更高。

### 暫定讀法

- **D 明顯領先（94）**，且領先幅度大到不太可能被 spike 結果翻盤——除非 pybind11 綁定成本遠高於預期。
- **B 第二（75），預期可過 L0 Gate，但爬不上階梯**。它的 1 分落在「效能餘裕」而非 Gate——在階梯設計下，B 的問題不是不能開始，而是沒有成長空間：M4 的優化敘事在純 Python 上很快撞到天花板。
- **C（Numba）是意外的輸家（69），與純 C++ 同分**。它在演算法表達力上被 Numba 的 nopython 限制拖累（僅 2 分），卻又拿不到 C++ 的效能上限與能力建構價值——兩頭不到岸。這一格與初始直覺相左，值得在 spike 中優先驗證。
- **A 墊底（69）不是因為它差**，而是完成風險與迭代速度在「同時學演算法 + 學 C++20」的情境下懲罰很重。

### D 為何在能力建構上也拿 5 分

分階段設計讓兩個目標不再互斥：階段一（Python）保證專案能完成、演算法能學會；階段二（C++ 核心重寫）取得無人機領域 JD 要求的 C++ 實力。順序保證了階段二即使停擺，階段一仍是完整可展示的專案。附帶效益是那次重寫會產出 M4 所需的優化前後對比數據——stretch goal（數十毫秒）自然成為階段二的驗收目標。

## 7.5 Spike 實測結果（E1，2026-07-20）

**方法**：各候選實作一份 Space-Time A*（4-connected + wait、Manhattan 啟發式、
closed set on `(x,y,t)`）。節點順序以打包成單一 int64 的優先權鍵
（`f<<42 | g<<30 | x<<21 | y<<12 | t`）統一，確保跨語言展開順序完全一致。
測資為 128×128、20% 障礙的合成 octile 地圖（seed 42，非真實 movingai 資料集，
見程式碼註解說明），100 組長距離單代理人查詢。硬體：AMD Ryzen 7 3700X。
編譯：`g++ -std=c++20 -O2 -march=native`。

**有效性檢查（通過）**：四個實作皆回報 `expansions=353985`、`cost_sum=15395`。
展開數與解答成本完全一致，代表比較的是純粹的每次展開成本。

**精確度**：吞吐量有 **5–8% 的執行間變異**（重跑一次：`cpp-flat` 5.70M、`python` 235K）。
展開數與成本則完全可重現。下表數字應讀作量級與比率，不應解讀到個位數精度；
本文件的所有結論在此變異範圍內皆維持成立。

| 實作 | closed set 結構 | expansions/sec | 對 python 倍率 |
|---|---|---:|---:|
| `cpp-flat` | 扁平陣列 + generation stamp | **6,020,774** | 23.4× |
| `numba` | 扁平陣列 + generation stamp | **5,234,545** | 20.4× |
| `cpp-hash` | `unordered_set` | 3,982,712 | 15.5× |
| `python` | 內建 `set` | 256,761 | 1.0× |

### 三個推翻先前判斷的發現

**(1) §4 的先驗預測錯了，方向偏向高估 C++。** 原假設 CPython 約 10⁵、C++ 約 10⁷，
即約 100 倍差距；實測為 **23 倍**。CPython 側預測尚可（2.6×10⁵），C++ 側高估了約 2 倍。
此為 E3 推論被 E1 推翻的實例，正是證據等級制存在的理由。

**(2) Numba 達到 C++ 的 87%，遠優於預期。** 原給 C 方案「效能餘裕」3 分，
實測顯示它與最佳化 C++ 僅差 15%，應修正為 5 分。

**(3) Numba 的表達力代價經實測確認（原判斷正確）。** nopython 模式下無可用的
hash set，CPython 的 `set` 無法沿用，被迫改為扁平陣列；`heapq` 不支援，二元堆必須
手工實作於預配置陣列上。因此 numba 版在結構上等同 `cpp-flat` 而非 `python`——
「用 Python 寫」的優勢在熱點內並不存在。「表達力 2 分」維持，且證據等級升為 E1。

### G1 Gate 判定：B 出局（E1）

以每次單代理人重規劃 3,540 次展開（353,985 ÷ 100）估算，100 agents 的**初始規劃**
（尚未進入任何衝突解決）所需展開數為 354,000：

| 候選 | 初始規劃耗時 | L0 預算 1000 ms |
|---|---:|---|
| `cpp-flat` | 59 ms | 通過，餘 94% 預算供衝突解決 |
| `numba` | 68 ms | 通過，餘 93% |
| `python` | **1378 ms** | **未通過——連 CBS 都還沒開始就已超支** |

**B（純 Python）在 100 agents 下未通過 L0，且此結論不需要任何 CBS 建模即成立。**
Gate 的設計目的在此兌現。

### 尚未取得：CBS 高階節點數

延遲公式的左因子（每次求解的總展開數）仍為未知，取決於衝突樹展開的節點數，
而該數量隨場景密度劇烈變化。本 spike 只確立了右因子。以 numba 的 5.2M/s 計算，
L0 的 1000 ms 預算約可支撐 1,470 次重規劃，扣除 100 次初始規劃後餘約 1,370 個
CT 節點——**這是否足夠 100 agents，必須實測 CBS 才能回答**，列為後續工作。

### 修正後分數

僅準則 #1 的 C 方案由 3 改為 5（E1 依據），其餘不變：

| 方案 | 原分 | 修正後 |
|---|---:|---:|
| **D 混合** | 94 | **94** |
| C Python + Numba | 69 | **77** |
| B 純 Python | 75 | 75 |
| A 純 C++ | 69 | 69 |

**D 仍然勝出，但勝出的理由改變了。** Numba 既然已達 C++ 的 87%，階段二的 C++ 重寫
在**純效能**上不再是必要——它的正當性現在幾乎完全來自能力建構（無人機領域 JD 要求 C++）。
這個理由依然成立且權重不低，但應誠實記錄：若哪天能力建構目標消失，D 對 C 的優勢會大幅縮小。
